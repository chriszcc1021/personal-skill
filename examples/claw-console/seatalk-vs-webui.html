SeaTalk AI vs Web UI 平台.

同一个模型 / 不同载体 / 不同上限

① 维度对比

SeaTalk AI 对话

IM 内嵌 · 黑盒模式
  • +零切换：人已经在 SeaTalk，开窗即聊
  • +群内可 @，社交化使用
  • +移动端原生体验
  • −纯文本气泡，长代码/diff 被压扁
  • −工具调用黑盒，看不到执行过程
  • −无附件渲染（PDF/HTML/截图弱）
  • −历史检索差，翻记录到死
  • −无 session 概念，单线程
  • −无配置可视（模型/reasoning 全黑）

Web UI 平台

工作台 · 白盒模式
  • +工具调用可见，逐步可审计
  • +富媒体：附件 / 截图 / HTML embed
  • +代码高亮 / 长 diff 完整展示
  • +多 session 并行 + 子 agent 视图
  • +session 列表 & 搜索可回看
  • +/status / reasoning / model 切换
  • +不污染工作群消息流
  • ~需主动打开页面，无 push 入侵感
  • ~移动端体验弱于原生 IM

结论 —— 模型一样，载体决定上限。SeaTalk 适合「轻问答 / 顺手聊」， Web UI 适合「干活 / 调试 / 工程闭环」。两者不是替代关系，是同一个 agent 的两张脸。

③ 平台改进方向

01

移动端 PWA / 原生壳

当前移动浏览器体验是短板。做 PWA 或薄壳 App，补齐 push 通知 + 离线消息队列。

P0
02

SeaTalk ↔ Web UI 互通

SeaTalk 触发的任务能在 Web UI 接手继续，附件 / 上下文双向同步。把两张脸合成一个身份。

P0
03

工具调用回放

每个 tool call 可单步回放、diff 文件改动、重跑某一步，类似 Jupyter 单元格。

P0
04

Session 协作 / 共享

把一个 session 一键分享给同事只读 / 可评论，省去截图扔群的中间环节。

P1
05

子 agent 拓扑视图

当前 sessions_spawn 出来的子任务靠列表展示。需要 DAG / 树状可视化看依赖和阻塞。

P1
06

成本面板

实时 token / 折后金额 / 模型分布；每个 session 单独算账，老板要数能直接导出。

P1
07

Memory 编辑器

MEMORY.md / skill 文件直接在 UI 里编辑、diff、版本回滚，不用切 IDE。

P1
08

Skill 市场

Skill 一键安装 / 评分 / 私有仓库；当前靠手动 clone 到 ~/.openclaw/workspace/skills。

P1
09

语音输入 / TTS 默认开启

移动场景下打字慢，PTT 录入 + TTS 朗读关键回复，匹配 SeaTalk 已有的语音习惯。

P2
10

权限分层 / 审批流

对外发邮件 / 跑生产命令走二次确认；目前靠 prompt 自觉，企业场景需要硬约束。

P2
11

多模态画布

白板式 canvas：拖文件 / 截图 / 链接进来，agent 自动理解上下文，不再线性 chat。

P2
12

本地代理 / 内网穿透托管

现在还要手开 cloudflare tunnel。平台直接管理 expose / 域名 / 鉴权，一键发布。
