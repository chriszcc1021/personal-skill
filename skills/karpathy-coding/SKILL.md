# SKILL: Karpathy 风格代码开发约束

## 什么时候用

**触发条件（命中任何一条都强制走流程）：**
- 用户让你修改一个已经写过的文件（`build_html.py` / HTML / CSS / 配置）
- 用户用了"调一下 / 改一下 / 修一下 / 加一个 / 删一个"这类词
- 上一轮你刚改过这个文件，本轮要继续动它
- 你打算用 `write`（整文件覆盖）而不是 `edit`（局部替换）

**不触发：**
- 从 0 写新文件
- 用户要求的就是大重构

---

## 4 条核心原则（来自 CLAUDE.md / Karpathy 风格）

### 1. 改前先思考，把模糊处问清楚
- **不要假设，不要藏起困惑，把权衡点摊开**
- 如果有多种解读，**列出来让用户挑**，不要默默选一个
- 如果有更简单的方案，**说出来**
- 不明白就停下来问

### 2. 简洁优先
- **解决问题的最少代码，不写投机性的东西**
- 没要求的功能不加
- 一次性代码不抽象
- 不预留"灵活性"
- 200 行能写成 50 行就重写

### 3. 外科手术式修改（最容易翻车的，必背）
- **只动你必须动的地方**
- 不要"顺手优化"相邻代码、注释、格式
- 不要重构没坏的东西
- 跟现有风格保持一致，哪怕你觉得别的写法更好
- 看到无关死代码，**提一句但别删**
- **测试**：每一行改动都能追溯到用户的原始要求

### 4. 目标驱动 + 验证
- 把任务转成可验证的目标
- 多步任务先列计划：`1. 步骤 → 验证什么；2. 步骤 → 验证什么`
- 弱目标（"让它工作"）= 不停回头找用户对齐
- 强目标（"测试通过"） = 你能独立闭环

## 强制工作流

### 改前：列清单（**永远第一步**）

在动键盘之前，**先 grep 一遍这个文件之前改过的关键字段**，列出"绝不能误删"的清单。例：

```bash
# 改 build_html.py 前
grep -nE '"span"|viewBox|hero-stamp|Doto|width:100%|UPDATED' build_html.py
```

输出"保留清单"贴给用户/写进当前 turn：
```
保留清单（改完必须还在）：
- "span":True 不在数据里（已删，别加回来）
- SVG 有 style="width:100%" （撑满容器）
- <link> 含 family=Doto （否则字体退化）
- hero-stamp 红色呼吸点
```

### 改时：用 `edit` 不用 `write`

- **能 `edit` 精确替换的，绝不 `write` 整文件覆盖**
- 多处改动 → 多个 `edit` 调用，每个只替换 5-20 行
- 如果必须大段重写，**先在 diff 心里跑一遍**：保留清单的每一项有没有被覆盖？

### 改后：跑回归探针

每次改完，**固定跑一遍探针脚本**（项目特定的检查清单），确认旧修复没回退：

```bash
# 示例（针对 fastpublish report）
grep -c 'class="card span2"' report.html     # 应该 = 0
grep -c 'preserveAspectRatio' report.html    # 应该 ≥ 1
grep -c 'family=Doto' report.html             # 应该 = 1
grep -c 'UPDATED.*GMT+8' report.html         # 应该 ≥ 1
```

探针失败 = 不要交付，回去再改。

---

## 不要做的事

| ❌ | ✅ |
|---|---|
| 整文件 `write` 一遍 | 多个 `edit` 局部替换 |
| 顺手把相邻的代码也"优化"了 | 只改用户要求的那几行 |
| 写完直接交付 | 先跑回归探针 |
| 模糊任务直接动手 | 先把"成功标准"写下来 |
| 自己默默选一个方案 | 列 A/B 让用户选 |

## 反例：本 skill 诞生的原因

> 2026-05-15：用户让加 hero-stamp + 改字体，我用大段重写把 Charles 卡的 `"span":True` 和 SVG `width:100%` 一起回退了。两次类似翻车。
>
> **教训**：每次改 `build_html.py` 必须先 grep "保留清单"，改完跑回归探针。
>
> **结论**：写这个 skill，固化流程。
